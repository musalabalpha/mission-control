# Mission Control 2.2.0

Mission Control 2.2.0 is a security and workspace-isolation release. It rolls the post-2.1 work into a reviewed baseline for self-hosted operators while keeping the existing trunk-based OSS workflow.

## Highlights

- Native workspace brand and isolation fields, backed by migration `052`
- Strict workspace boundaries across request, scheduler, event, audit, gateway, filesystem, and background-dispatch paths
- Safer host CLI dispatch with explicit tool, budget, and workspace-root constraints
- Hardened privileged mutations for gateway control, runtime setup, users, skills, backups, updates, approvals, and host maintenance
- Tailwind CSS v4 migration with visual verification across core surfaces and themes
- Stronger protected CI with immutable action-pin and standalone artifact-boundary checks
- Structured OSS issue and PR intake with clearer provenance, risk, and verification contracts

## Security posture

The security work in this release concentrates on authority boundaries rather than UI-only controls. Ambiguous workspace ownership fails closed, sensitive mutations are throttled and strictly validated, command output is bounded or redacted, audit records avoid host paths and secrets, and privileged client calls reject failed API responses.

The release-preparation audit found no known high-severity dependency vulnerabilities. The protected quality gate covers API parity, lint, typecheck, unit tests, production build, standalone artifact boundaries, and the full Playwright suite.

## Upgrade notes

1. Back up the Mission Control database and runtime configuration.
2. Upgrade from a protected `v2.2.0` tag or corresponding immutable container digest.
3. Run migrations before enabling strict isolation on existing workspaces.
4. Review `MC_WORKSPACE_ROOT`, trusted proxies, gateway credentials, and host-runtime permissions.
5. Re-test downstream theme customizations against Tailwind CSS v4 CSS tokens.

Workspace classification and isolation metadata do not grant execution authority. The declarative approval-policy engine remains future roadmap work.

## Compatibility

- Node.js 22 or newer remains required.
- Existing shared workspaces remain supported.
- The release does not intentionally remove public API routes.
- Custom Tailwind configuration is the primary downstream migration surface.

## Verification

The release commit must pass the repository's protected `quality-gate` and screenshot-drift checks before tagging. Tags are created only from the merged `main` commit.
